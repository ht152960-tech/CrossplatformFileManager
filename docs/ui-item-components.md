# UI Item Components

## 当前阶段执行原则

- 本文档是长期 UI 条目规范，不代表每次修改都要一次性实现全部组件。
- 当前阶段只做首页宽屏版完成后的组件抽取准备。
- 先抽 Design Tokens，再抽基础视觉容器，再抽标签和按钮。
- 暂缓抽完整业务组件 `FileItem` / `RecommendedFileItem` / `RecentFileItem`。
- 暂缓重构中屏、窄屏和其他页面。
- 首页宽屏版视觉效果视为基准，不允许破坏。
- 如果发现现有页面与本文档不一致，先记录 TODO，不要顺手大改。

已建立：

- `TaggoColors`
- `TaggoRadius`
- `TaggoSpacing`
- `TaggoTypography`
- `TaggoAlpha`

## 屏幕尺寸

- `W`：宽屏 / `expanded`
- `M`：中屏 / `medium`
- `C`：窄屏 / `compact`

规则：

- 宽屏可显示更多辅助信息。
- 中屏保留核心信息，减少辅助信息。
- 窄屏优先保留识别和操作必需信息。
- 不同屏幕可以改变布局密度，但不能改变条目语义。
- 空间不足时，优先隐藏低优先级信息，避免压坏布局。

## 通用文件条目 `FileItem`

适用：

- 全部文件列表
- 搜索结果列表
- 最近添加条目的基础形态
- 推荐条目的基础形态
- 文件详情页中的相关文件列表

必须显示：

- 文件图标 / 缩略图
- 文件名.扩展名
- 添加时间、文件大小、标签属于常规信息。
- 不同变体可以根据空间和页面语义选择显示常规信息。

可选显示：

- 文件类型
- 最近打开时间
- 文件路径
- 更多操作按钮
- 标签数量
- 文件状态提示

不应显示：

- 假数据
- 未实现功能的按钮
- 与当前页面无关的复杂统计
- 无法解释的推荐分数

点击行为：

- 点击文件图标 / 缩略图：直接打开文件。
- 点击条目其他区域：进入文件详情页。
- 该规则适用于 `W`、`M`、`C`，不同页面不应随意改变。

缩略图遮罩规则：

- 视频文件：播放键遮罩。
- 音频文件：播放或音频提示遮罩。
- 图片文件：图片类型提示或轻量遮罩。
- 普通文档：使用文件类型图标。
- 遮罩应轻量，不破坏缩略图识别。

响应式规则：

- `W`：显示完整信息，可补充路径、最近打开时间或操作按钮。
- `M`：保留文件图标 / 缩略图、文件名.扩展名、添加时间、文件大小、标签；空间不足时收敛标签数量或标签数量提示。
- `C`：优先保留文件图标 / 缩略图、文件名.扩展名、添加时间；文件大小、标签可以隐藏或收纳。

## 最近添加条目 `RecentFileItem`

本质：

- `RecentFileItem` 是 `FileItem` 的紧凑变体，不是全新类型。

适用：

- 首页最近添加卡片
- 其他需要展示最近添加文件的小型列表区域

必须显示：

- 文件图标 / 缩略图
- 文件名.扩展名
- 添加时间

- `RecentFileItem` 作为紧凑变体，不应被强制显示完整 `FileItem` 的全部常规信息。

可选显示：

- 文件大小
- 文件类型

不应显示：

- 文件路径
- 推荐理由
- 复杂操作按钮
- 过多标签
- 与最近添加无关的统计信息

视觉要求：

- 每一行外层应填满父容器宽度。
- 所有条目的视觉长度应一致。
- 右侧时间区域应有稳定宽度。
- 文件名过长时应截断，不能撑破布局。
- 背景和边框应轻，不要像厚重输入框。

点击行为：

- 点击文件图标 / 缩略图：直接打开文件。
- 点击条目其他区域：进入文件详情页。

## 推荐条目 `RecommendedFileItem`

本质：

- `RecommendedFileItem` 是 `FileItem` 的增强变体。
- 它必须展示“为什么推荐这个文件”。

适用：

- 首页智能推荐主面板
- 推荐页主列表
- 未来与推荐算法相关的文件列表

必须显示：

- `W` / `M`：文件图标 / 缩略图、文件名.扩展名、推荐理由、标签
- `C`：文件图标 / 缩略图、文件名.扩展名、推荐理由

可选显示：

- 文件大小
- 最近打开时间
- 添加时间
- 推荐强度说明
- 更多操作按钮

不应显示：

- 无法解释的神秘分数
- 假推荐理由
- 未实现的 AI 分析说明
- 过于复杂的统计内容

推荐理由应尽量来自真实规则或已有数据，例如：

- 最近常用
- 与当前时间规律匹配
- 打开某文件后经常接着打开
- 长时间未打开但符合周期
- 与最近打开内容相关

如果无法提供明确理由，应使用保守文案，不得伪造复杂解释。

点击行为：

- 点击文件图标 / 缩略图：直接打开文件。
- 点击条目其他区域：进入文件详情页。
- 如果存在明确“打开”按钮，点击该按钮也应直接打开文件。

视觉原则：

- 可以比普通文件条目更突出，但不能脱离文件条目的视觉家族。

## 标签条目 `TagItem` / `TagChip`

`TagItem` 与 `TagChip` 分开约束。

### `TagItem`

适用：

- 标签列表
- 首页标签入口
- 标签管理页

必须显示：

- 标签字符串
- 有多少个文件拥有这个标签

可选显示：

- 删除按钮
- 编辑入口
- 颜色标记
- 标签状态

不应显示：

- 搜索关键词伪装成正式标签
- 未保存的临时条件
- 与标签无关的统计

点击行为：

- 点击正式标签：进入标签筛选或标签详情。
- 删除按钮只在可编辑场景出现。
- 编辑入口只在明确支持编辑的页面出现。

视觉原则：

- 正式标签表示长期存在的数据，视觉上应更稳定、更肯定。
- 正式标签可使用较实的背景或较稳定的紫色层级。

### `TagChip`

适用：

- 文件条目
- 文件详情
- 标签摘要

必须显示：

- 标签名

默认不显示：

- 文件数量

可选显示：

- 删除按钮
- 编辑入口
- 颜色标记
- 标签状态

不应显示：

- 搜索关键词伪装成正式标签
- 未保存的临时条件
- 与标签无关的统计

点击行为：

- 点击正式标签：进入标签筛选或标签详情。
- 删除按钮只在可编辑场景出现。
- 编辑入口只在明确支持编辑的页面出现。

视觉原则：

- 正式标签表示长期存在的数据，视觉上应更稳定、更肯定。
- 正式标签可使用较实的背景或较稳定的紫色层级。

## 搜索条件 / 临时搜索标签 `SearchTokenChip`

本质：

- `SearchTokenChip` 表示用户在搜索过程中临时形成的搜索条件。
- 它不是正式标签，不会写入文件标签数据。

适用：

- 搜索框下方
- 搜索条件区域
- 用户输入关键词后生成的临时搜索标签列表

必须显示：

- 搜索关键字

建议显示：

- 删除 / 关闭按钮
- “关键词”或搜索图标提示

不应显示：

- 文件数量，除非搜索结果已明确计算
- 正式标签的文件数量
- 让用户误以为它已被保存为标签的样式

点击行为：

- 点击关闭按钮：移除该搜索条件。
- 点击主体：可以聚焦或高亮该条件，但不进入标签详情。
- 不得把临时搜索关键词写入文件标签数据。

规则：

- 搜索条件 / 临时搜索标签必须统一带关闭按钮。
- 关闭按钮统一放在标签右上角。
- 点击关闭按钮时，不应触发标签主体点击行为。
- 关闭按钮视觉上要清楚，但不能过重。
- 关闭按钮样式应与当前项目中采用的右上角删除样式保持一致。
- 不得在不同页面中一会儿放右侧、一会儿放右上角。
- 临时搜索标签必须始终显示关闭按钮。

与正式标签的关系：

- 同属一个视觉家族，但必须有区别。
- 正式标签：更实、更稳定，代表长期保存的数据。
- 临时搜索标签：更轻、更透明，带关闭按钮，代表当前搜索条件。

## 效果图使用原则

- 效果图只作为视觉风格参考。
- 不得复制其中假数据、固定统计值、固定文件名或未实现功能。
- 颜色、圆角、间距应沉淀为 token，而不是在页面中硬编码。

## 文件条目中的标签显示规则

- 最多直接显示 3 个标签。
- 超过 3 个时，只显示前 3 个。
- 剩余未显示的标签必须用数量提示说明，例如 `+2`、`还有 2 个`、`+2 个标签`。
- 不得因为标签过多撑破条目布局。
- 不得在宽屏、中屏、窄屏中使用完全不同的标签数量规则。
- 窄屏空间不足时，可以只显示 1-2 个标签，但仍必须用剩余数量提示说明未显示数量。
- 标签排序优先使用当前已有逻辑；如果没有明确排序逻辑，保持原始顺序。

## 文件类型条目 `FileTypeItem`

适用：

- 首页文件类型卡片
- 文件页类型筛选
- 类型统计区域

必须显示：

- 类型图标
- 类型名称
- 有多少个文件属于这个类型

可选显示：

- 占比条
- 占比百分比
- 类型说明

不应显示：

- 没有真实来源的统计
- 复杂图表，除非页面明确需要
- 与类型筛选无关的操作

点击行为：

- 进入该类型的筛选结果，或在当前页面应用类型筛选。
- 具体行为应根据页面已有逻辑决定，但不能在不同页面中表现得互相矛盾。

视觉原则：

- 文件类型条目应比文件条目更轻。

## 卡片 Footer `DashboardCardFooter`

适用：

- 最近添加卡片底部
- 标签卡片底部
- 文件类型卡片底部

必须显示：

- 明确入口文案，例如 `查看全部 →`

可选显示：

- 轻量箭头图标
- 轻量 hover 状态

不应显示：

- 额外统计
- 未实现功能按钮
- 过亮的独立底色条
- 与卡片主体割裂的亮色框

点击行为：

- 跳转到对应完整页面，或执行当前已有的查看全部逻辑。

视觉原则：

- footer 是卡片自然的底部区域，不是额外贴上去的一块亮色按钮。
- 三张入口卡片的 footer 风格必须统一。
- footer 背景应弱化。
- footer 与卡片主体应自然连接。
- 不应形成突兀亮框。

## 组件变体建议

- `FileListItem`
  - `Compact`：最近添加、小卡片、小列表
  - `Regular`：全部文件、搜索结果
  - `Featured`：推荐主面板或重点文件
- `RecommendedFileItem`
  - `Compact`：窄屏推荐
  - `Regular`：普通推荐列表
  - `Featured`：宽屏主推荐面板
- `TagChip`
  - `Regular`：正式标签
  - `Editable`：文件详情页可编辑标签
  - `SearchToken`：临时搜索条件标签
- `FileTypeSummaryItem`
  - `Compact`：首页文件类型卡片
  - `Regular`：类型筛选或完整类型页
- `DashboardCardFooter`
  - `Default`：普通查看全部入口

## 后续重构规则

- 页面重构时，优先复用上述基础组件。
- 不允许为相同语义重新写一套卡片、条目、标签、按钮、空状态样式。
- 如果现有组件不满足需求，先记录差异，再决定扩展组件、新增 variant，或保持页面局部实现。
- 不得为了复用而把业务逻辑塞进基础组件。
- 业务组件如 `FileItem` / `RecentFileItem` / `RecommendedFileItem` 暂缓抽取，等中屏、窄屏和其他页面形态稳定后再判断。

## 统一点击行为

统一点击行为

点击文件图标 / 缩略图：直接打开文件。
点击条目其他区域：进入文件详情。
不再新增独立“打开”按钮。
如果存在历史遗留打开按钮，应逐步移除。
如果存在“更多”按钮，点击更多按钮只打开操作菜单，不应触发进入详情。
不同页面不得随意改变这套基础行为。

统一原则：

文件打开入口只有一个：

文件图标 / 缩略图。

不允许：

再新增 Open 按钮
再新增“打开”按钮
在条目右侧重复提供打开入口
在推荐条目和普通文件条目中采用不同打开规则

- 深蓝黑背景
- 黑紫 / 暗紫卡片

- 白色作为主文字

- 主强调紫：`#8B5CFF`
- 亮紫高光：`#A970FF`
- 按钮 / 发光紫：`#9C63FF`

辅助强调色：

- 蓝紫进度条 / 次强调蓝：`#5B7CFF`

## 颜色使用原则

1. 页面背景以深蓝黑和黑紫为主。
2. 卡片背景保持暗紫黑，不要偏蓝。
3. 主操作按钮使用紫色。
4. 选中态使用紫色系，不使用高饱和科技蓝。
5. 统计和类型图表可使用蓝紫、粉紫、橙色辅助，但不能喧宾夺主。
6. 正文使用白色。
7. 说明文字使用灰白。
8. 边框和高光应克制，避免霓虹感过强。
9. footer 不应使用过亮背景。
10. 列表项背景不应比卡片主体更抢眼。

## 未来实现约束

1. 不得在不同页面中随手重新画一套相同语义的条目。
2. 优先复用已有组件。
3. 如果已有组件不满足需求，应扩展 variant，而不是复制一份新样式。
4. 如果必须新增组件，应先说明为什么不能复用旧组件。
5. 所有条目应属于同一套黑紫 / 暗紫视觉系统。
6. 宽屏、中屏、窄屏只能改变布局密度和低优先级信息显示，不能改变条目核心语义。
7. 页面重构前，应先检查本页面使用的条目是否符合本文档。
8. 页面重构后，应自查是否出现同类条目视觉不一致的问题。
9. 不得写死假数据。
10. 不得改变真实统计来源。
